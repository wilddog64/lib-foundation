# Retrospective — v0.4.0

**Date:** 2026-06-22
**Milestone:** Absorb lib-acg as the optional `scripts/lib/acg/` browser-automation module
**PR:** #32 — merged to main (`aed8c56`)
**Participants:** Claude, Codex, Copilot

## What Went Well

- **Root-caused the viewport regression with a controlled experiment.** A byte-for-byte lib-acg
  copy isolated the cause of the recurring "Element is outside of the viewport" failure: both the
  hoisted-root and module-local `bin/` layouts resolve `REPO_ROOT` to the same absolute path — the
  only runtime difference was the **working directory at launch** (repo root vs `scripts/lib/acg`).
  Running from the module dir, matching upstream lib-acg, is what the live Playwright flow needs.
- **All 12 Copilot findings fixed and resolved inside the PR.** 2 Makefile (`f3f2fbd`) + 10
  imported-code hardening (`cbe203f`); every thread replied with the fix SHA and resolved (0 left).
- **Live validation, not just static gates.** `make credential-test PROVIDER=aws` passed both
  scenarios (sandbox-exists reuse + sandbox-delete/restart) ending in `sts:GetCallerIdentity OK`,
  confirming the CDP host/port threading and the `gcp.sh` no-`source` parser did not regress the
  live flow.
- **Clean module isolation.** Opt-in Node (`npm ci` in the module dir), sources `../system.sh` for
  `_run_command` (no vendored foundation), core sourcing stays zero-node, independent `acg` CI job.

## What Went Wrong

- **The Phase 1 follow-up (`e6cecef`) hoisted `bin/` to the repo root and regressed the live flow.**
  It cost a debugging cycle; reverted in `cf3ad7a`. Lesson: browser-automation entry points are
  sensitive to launch working directory — preserve the upstream layout exactly.
- **The verbatim import carried a pre-existing latent bug.** `acg-credential-test:277` calls an
  undefined `_sts_valid` → exits 127 → the `!` makes it always-true → a needless sandbox
  delete/restart fires on every happy-path run even when the freshly-extracted creds are valid.
  Surfaced only in the live run, not in CI. Tracked as a follow-up (one-line fix).
- **CHANGE.md versioning drift.** v0.3.18–v0.3.20 are tagged but were never cut from `[Unreleased]`,
  and the v0.4.0 absorption is still sitting under `[Unreleased]` rather than a `## [0.4.0]` section.
  The release was not version-stamped at merge.

## Decisions Made

- **Absorb, don't subtree.** lib-acg becomes a self-contained optional module (clean tree-copy from
  `7708ea31`, v0.1.9), retiring the 3-level subtree chain and the standalone repo.
- **`bin/` stays module-local.** The Makefile `cd`s into `scripts/lib/acg` before invoking the
  entry points so the live browser flow runs with the module as its working directory.
- **Hardening folded into the absorption PR** rather than deferred — the verbatim-fidelity principle
  is relaxed for genuine defects when the user opts in, with defaults held constant (127.0.0.1:9222).
- **Toolchain shift noted:** Google retired gemini-cli in favor of agy-cli (Antigravity); the lib-acg
  CDP automation will need a retarget — scoped separately.

## Theme

v0.4.0 folds the ACG browser-automation tooling into lib-foundation as a single self-contained,
opt-in module — collapsing a brittle 3-level subtree chain into one tree-copy that sources the
foundation directly. The hard-won lesson of the milestone is mundane but expensive: browser
automation depends on its launch working directory, so fidelity to the upstream layout is
load-bearing, not cosmetic. Static gates were necessary but not sufficient — only the live sandbox
run proved the absorption (and exposed a latent `_sts_valid` bug the import inherited).
