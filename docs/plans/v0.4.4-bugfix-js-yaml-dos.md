# Bugfix: v0.4.4 — close js-yaml DoS advisory (GHSA-h67p-54hq-rp68)

**Repo:** `lib-foundation` (this repo — the canonical upstream, NOT a subtree consumer)
**Branch:** `feat/v0.4.4`
**Files:** `scripts/lib/acg/package-lock.json`, `CHANGE.md`

---

## Problem

Dependabot flags **GHSA-h67p-54hq-rp68** (severity **medium**): *"JS-YAML: Quadratic-complexity
DoS in merge key handling via repeated aliases"*, vulnerable range `< 3.15.0`, first patched
`3.15.0`.

The `scripts/lib/acg` npm project resolves `js-yaml` at **3.14.2** (a **dev-only transitive**
dependency pulled in by the jest/babel test toolchain — it never runs on any ACG runtime path).
The alert surfaces in the consumer repo `k3d-manager`, where this project is vendored via the
lib-foundation subtree at `scripts/lib/foundation/scripts/lib/acg/package-lock.json`. Because that
file is subtree-managed, the fix **must be made here upstream first**, then subtree-pulled into
k3d-manager — editing the k3d-manager copy directly would be clobbered on the next subtree pull.

**Root cause:** the committed `package-lock.json` pins the transitive `js-yaml` at `3.14.2`, one
patch below the fixed `3.15.0`. The direct declared range is already `^3.13.1`, which permits
`3.15.0` — so only the lockfile needs to move.

---

## Reproduction

```bash
grep -A1 '"node_modules/js-yaml"' scripts/lib/acg/package-lock.json
# Actual (baseline):  "version": "3.14.2",   ← vulnerable
# Expected (fixed):   "version": "3.15.0",
```

## Fix

Regenerate the lockfile's `js-yaml` entry with npm — do **not** hand-edit JSON. Run this from the
repo root:

```bash
npm update js-yaml --package-lock-only --prefix scripts/lib/acg
```

This produces **exactly** the 3-line change below (verified via prototype 2026-07-13). `package.json`
is **unchanged** (the bump stays within the existing `^3.13.1` range), and `js-yaml` 3.15.0 carries
the same `argparse`/`esprima` deps as 3.14.2, so there is **no cascade** — only the single `js-yaml`
node moves.
